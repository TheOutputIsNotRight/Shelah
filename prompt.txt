The website is called Shelah (Egyptian slang for a group of people). It is made to solve the issue of not knowing where to go in an outing and the fact that each user has different requirements for what the place can be. It should reconcile those requirements by asking each user about them and recommending places that fit all of those users' requirements. The requirements should be Distance from where they live (where they live should be taken from a map prompt to the user where the website also tries to use GPS to make it easier), Popularity (Niche or Mainstream), Rating, Budget and Location Type (generate multiple location types like coffee shops, restaurants, museums, games etc). The database should be expected to have these places stored with all that information for each place. Users can sign up with their email and password. Users can add each other as friends. Users can create outings with an initial list of invitees from their friends list. Each outing can have a type from three: Open (anyone can join), Closed (no one extra can join) and Restricted (people can be invited but only after everyone who is already invited to this group has accepted them to be invited). Each outing has a name and a set date for when it will happen that the initial creator/admin will set. After the initial creator has made the outing invites will be sent to each friend they chose to invite and a link will be shown that is the format of "[the website's domain]/outings/[outing id]". That link can be opened by anyone, if they're not logged in and it's restricted/closed or they're logged in and they are not invited then opening the link says it's a restricted outing. If the outing is open or they're logged in and invited then it would say they are invited to the [outing's name]. Once the user is accepted into the outing then opening its link will show a page with the outing's name, date and its type. In a list it will show each user invited to the outing's 1) whether they accepted the invite yet or not 2) Whether they put their requirements or not yet. That page will let the user edit their requirements for each type as described above and will also allow the user to look at the recommended places. The recommended places will follow all user's requirements and will be sorted according to the number of votes. Each user in the outing can vote to as many places as they want. The database should be structured as the website's functionality needs.



Each page should be as will be described:

1) The home page should have 4 buttons:

a) Add friend (shows a prompt where they can enter a friend's email)

b) Outings (redirects to a page summarizing all outings the user is part of and clicking on one of them redirects to the outing's page)

c) Sign up/Sign in

d) Create outing (redirects to create outing page)



2) The create outing page should have 5 buttons

a) Confirm (redirects to the newly created outing's page)

3) Outing page shows a link to the outing that users can share with other users. The outing's name, date and type. It should also show a list of all invited users, their acceptance status and whether they finished inputting their requirements. It should have 3 buttons.

a) Invite friend (if the group is open/restricted but not closed)

b) Edit requirements (shows user's requirements and allows the user to edit them each)

c) View recommendations (shows the list of recommended places, each place has its thumbnail picture, its distance from the user, its rating, its popularity, its price, its location type, a votes counter and an upvote button)

If the user has not yet accepted the invite to the outing then the page should instead show the page as previously described with an Accept invite if they can accept.



4) Outings page shows the summarized information about all the user's current outings, the outing name, date, type, the invited users and a View button to visit the outing's own page.
